# Contributing to Math-Problems

First off, thank you for considering contributing to Math-Problems! It's people like you that make this journey into computational mathematics and chaos theory such a great resource for students and developers.

## How Can I Contribute?

### 1. 🐛 Reporting Bugs

If you find a bug in a simulation or an algorithm, please open an issue on GitHub!

* **Use a clear and descriptive title.**
* **Describe the exact steps** which reproduce the problem.
* **Provide specific examples** to demonstrate the steps.

### 2. 💡 Suggesting Enhancements

Have an idea for a new algorithm, a better way to visualize a Mandelbrot set, or a new evolutionary parameter for GenePool?

* Open an issue to discuss your idea before writing code. This ensures your work aligns with the project's goals.

### 3. 👩‍💻 Submitting a Pull Request

If you are ready to write code, follow these steps:

1. **Fork the repository.**
2. **Create a new branch:** `git checkout -b feature/your-feature-name`
3. **Write your code.**
    * *If you are adding a Python script:* Ensure it runs smoothly and include a `requirements.txt` if needed.
    * *If you are adding C++ code:* Ensure your `CMakeLists.txt` is updated and builds correctly.
4. **Update the Documentation.** If you add a new algorithm, please add a brief explanation to the appropriate section in the Astro documentation (`docs/src/content/docs/`).
5. **Commit your changes:** `git commit -m "Add some feature"`
6. **Push to the branch:** `git push origin feature/your-feature-name`
7. **Submit a Pull Request.**

## 🎓 For Students

We especially welcome contributions from students learning algorithms and chaos theory! If you are tackling a "Good First Issue" or just want to add a basic sorting algorithm visualization, don't be afraid to submit a PR. We will review it and help you get it merged.

## 📝 Coding Style

* Try to keep your code well-commented. Mathematical algorithms can be dense, so explain *why* the code does what it does, not just *what* it does.
* If you are writing interactive visualizations, strive for real-time performance (e.g., 60 FPS for OpenGL applications).

Thank you for being part of the journey!
